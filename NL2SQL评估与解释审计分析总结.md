# NL2SQL 评估与结果解释审计 — 分析总结

> 基于 `eval_result.json` 和 `explain_eval_result.json` 的实测数据

---

## 一、NL2SQL 生成评估（eval）

### 1.1 总体结果

| 指标 | 数值 |
|:-----|:-----|
| 总用例数 | 17（12 正例 + 5 负例） |
| 通过 | **5** |
| 失败 | **12** |
| 通过率 | **29.4%** |

### 1.2 Schema 提示对比（12 正例）

| 对比结果 | 数量 | 说明 |
|:---------|:-----|:-----|
| Strong 改进 | **2** | P02、P09：strong 生成正确列名，weak 幻觉 `id` |
| Strong 退化 | **1** | P01：strong 简化查询导致结果列不匹配 |
| 均通过 | **1** | P12：两种模式都正确 |
| 均失败 | **8** | 大部分是 empty_sql |

**结论**：加入完整列名清单的 Strong schema 提示**确实有改进效果**（2/3 有差异的用例中是正向改进），但改进空间有限——主要瓶颈不是列名，而是 LLM 对 PostgreSQL 自定义函数的支持。

### 1.3 错误类型统计

| 错误类型 | 次数 | 占比 | 根因分析 |
|:---------|:-----|:-----|:---------|
| **empty_sql** | **8** | **66.7%** | DeepSeek 无法理解 `api.*` 自定义函数，拒绝生成 SQL |
| aggregate_misuse | 1 | 8.3% | P10：混淆 WHERE/HAVING，GROUP BY 不一致 |
| missing_join_condition | 1 | 8.3% | P10：多表连接遗漏条件 |
| llm_call_failed | 1 | 8.3% | Weak 模式下 JSON 解析失败（重试耗尽） |

### 1.4 关键发现

#### 🔴 发现一：自定义 API 函数是 NL2SQL 盲区

失败用例中 **P03、P04、P06、P07、P11** 共 5 个用例的 Gold SQL 依赖 `api.find_airfoils_by_condition()`、`api.compare_airfoils_at_reynolds()` 等 PostgreSQL 自定义函数，DeepSeek 对这些函数**完全无法生成 SQL**，一律返回空字符串。

**建议**：在报告中将此作为 "大模型对数据库专有函数的理解边界" 进行讨论。可对比：如果 Gold SQL 改为直接表查询（如 P08 的做法），准确率会显著提升。

#### 🔴 发现二：列名幻觉是 weak 模式的系统性问题

Weak 模式下，DeepSeek 系统性将 `version_id`、`airfoil_id`、`user_id` 简化为 `id`：
- P01 weak: `av.id`（应为 `av.version_id`）
- P12 weak: `airfoil.id` + `av.id` + `u.id`（三处皆错）

Strong 模式通过提供精确列名清单**完全消除**了此类错误。

#### 🟡 发现三：N05 版本语义审计漏报

N05 预期：模型直接查 `airfoil_version` 而不经过 `v_current_airfoil_version`，应被审计拦截为 `needs_fix`。

实际：审计引擎**放行**了该 SQL（`approved`），说明当前版本语义校验规则**未生效**。这是审计引擎的一个缺陷，需修复。

#### 🟢 发现四：负例拦截总体有效

5 个负例中，3 个被正确拒绝（N01 恶意注入、N04 敏感数据），2 个（N02、N03）被拒绝但预期是 `needs_fix` 而非 `rejected`——审计比预期更严格，属于偏安全侧的"过度防御"。

## 二、结果解释审计（explain-eval）

### 2.1 总体结果

| 指标 | 数值 |
|:-----|:-----|
| 总用例 | 7 |
| 成功生成解释 | **4** |
| 因 empty_sql 失败 | **3** |
| 解释可用率 | **57.1%** |

### 2.2 逐条分析

| 用例 | 状态 | 解释质量 |
|:-----|:-----|:---------|
| **E01** 升阻比Top10 | ❌ empty_sql | 因 P03 SQL 未生成，无结果可解释 |
| **E02** 分组统计 | ✅ 成功 | 解释引用了具体数值：`real` 类 2907 条（12 异常）、`synthetic` 类 20000 条（177 异常），**基于结果、无幻觉** |
| **E03** 异常翼型列表 | ✅ 成功 | 列出了 NACA_0021（6条异常）、NACA_63(3)-218 等，解释了 `total_anomaly_hint` 含义，**符合工程常识** |
| **E04** 曲线数据 | ❌ empty_sql | 因 P04 SQL 未生成 |
| **E05** 版本差异 | ✅ 成功 | 引用了 delta_cd 具体值，解释了版本间 Cd 差异的来源，**引用数据、不虚构** |
| **E06** 空结果对抗 | ✅ 成功 | 正确识别 NACA_9999 不存在，说明了"查询无结果"的原因，**未擅自补充信息** |
| **E07** 模糊问题 | ❌ empty_sql | "帮我找一个最好的翼型" — LLM 正确拒绝生成 SQL |

### 2.3 幻觉分析

在 4 个成功生成解释的用例中：

| 检查维度 | E02 | E03 | E05 | E06 |
|:---------|:---:|:---:|:---:|:---:|
| 解释基于结果（引用具体数值） | ✅ | ✅ | ✅ | ✅ |
| 符合工程常识 | ✅ | ✅ | ✅ | ✅ |
| 听起来合理但实际错误 | ❌ 无 | ❌ 无 | ❌ 无 | ❌ 无 |
| 数据不足时擅自补充 | ❌ 无 | ❌ 无 | ❌ 无 | ❌ 无 |

**结论**：在有数据可解释的场景下，DeepSeek 的结果解释**质量较高**，4 个用例均未发现幻觉。这得益于提示词中明确的约束（"必须引用数值、禁止补充不存在的信息"）。

## 三、综合结论与改进建议

### 3.1 可用于报告的素材

| 评分点 | 已有素材 |
|:-------|:---------|
| §9.1 NL2SQL 错误类型总结 | empty_sql（66.7%）、aggregate_misuse、missing_join_condition、field_name_error（weak 模式） |
| §9.1 Schema 提示改进分析 | Strong 消除列名幻觉（P02/P12），但无法解决 API 函数盲区 |
| §9.2 幻觉分析 | 4 例成功解释均无幻觉，约束提示词有效 |
| 批判性审计 | N05 审计漏报、API 函数盲区 — 体现"不盲信 AI"的批判思维 |

### 3.2 改进方向

1. **修复 N05 审计漏报**：版本语义检查规则在 `audit.py` 中需确认生效
2. **扩展测试用例**：增加不依赖 `api.*` 函数的直接表查询用例，对比准确率
3. **Schema 提示优化**：在 strong 模式中增加"禁止使用 `id` 作为列名"的显式约束
