> 选取 7 题中的 6 题

## 问题一：坐标点数据是否适合直接以 JSON 方式存储，为什么？

**结论：不适合。本系统采用行存储（每点一行），这是正确的选择。**

本系统 `coordinate_point` 表设计为：

```sql
CREATE TABLE coordinate_point (
    point_id    uuid,
    version_id  uuid,      -- FK → airfoil_version
    surface     text,       -- 'upper' / 'lower'
    point_order integer,
    x           numeric,
    y           numeric,
    is_deleted  boolean
);
```

**为什么不使用 JSON？**

| 维度 | 行存储（当前设计） | JSON 存储 |
|:-----|:------------------|:----------|
| **约束能力** | 可对 `point_order` 建唯一约束防止重复；可对 `x`/`y` 做范围 CHECK | JSON 内部字段无法施加数据库级约束 |
| **查询能力** | `WHERE surface='upper' ORDER BY point_order` 走复合索引，O(log n) | 需 `jsonb_array_elements()` 展开，无法利用 B-tree 索引 |
| **完整性** | `version_id` 外键保证引用完整性 | JSON 脱离引用约束，可能出现悬空版本 |
| **部分更新** | 修改单个点只需 UPDATE 一行 | 需整体替换整个 JSON 数组，无版本追踪粒度 |
| **数据治理** | 每个点可独立标记 `is_deleted`，配合触发器写 `change_log` | JSON 内修改无法触发行级触发器 |

本系统有 50,045 个坐标点，核心查询是"给定翼型当前版本，按 `point_order` 排序取上下表面轮廓"。行存储 + `(version_id, surface, point_order)` 复合索引使此查询只需一次索引扫描，JSON 方案则需要全表扫描 + 应用层解包 + 排序。

**设计反思**：JSON 在"一次读取、整体消费"的场景（如前端渲染）有便利性，但数据库的核心价值在于约束、索引和事务——JSON 存储完全放弃了这些能力。所以我们在数据库层用行存储，仅在应用层按需组装为 JSON 返回前端。

## 问题二：当数据规模扩大 100 倍后，当前模式和索引是否仍然适用？

**结论：核心模式适用，但需要在分区和物化视图层面做扩展。**

当前规模 → 100 倍后的预估：

| 表 | 当前 | ×100 后 | 风险评估 |
|:---|:-----|:--------|:---------|
| `airfoil` | 100 | 10,000 | 无压力，主键索引即可 |
| `airfoil_version` | 500 | 50,000 | `(airfoil_id, is_current)` 部分索引可覆盖 |
| `coordinate_point` | 50,045 | ~500 万 | ⚠️ 需分区 |
| `performance_record` | 22,907 | ~230 万 | ⚠️ 需分区 + 物化视图 |

**仍然适用的部分**：
1. **UUID 主键**：100 倍规模下自增 ID 也不会溢出，但 UUID 在分布式合并场景下仍有优势
2. **规范化模式**：3NF 设计避免了更新异常，100 倍数据下冗余代价更高，规范化更有价值
3. **覆盖索引策略**：`INCLUDE (cl, cd, l_over_d)` 覆盖索引在百万级仍能消除回表
4. **部分索引**：`WHERE is_current=true AND is_deleted=false` 过滤历史版本的比例不变（~80%），效果保持

**需要扩展的部分**：
1. **`coordinate_point` 按 `version_id` 做 HASH 分区**：500 万行时全表扫描不可接受，分区后可并行扫描
2. **`performance_record` 按 `reynolds_number` 做 RANGE 分区**：常用查询按 Re 筛选，分区裁剪显著减少扫描量
3. **物化视图增加并发刷新**：`mv_airfoil_stats` 在百万级数据下刷新耗时会增加，需改为 `REFRESH MATERIALIZED VIEW CONCURRENTLY`
4. **`pg_trgm` GIN 索引**：模糊搜索在 10,000 翼型下 GIN 索引仍然有效，但可能需要调整 `gin_fuzzy_search_limit`

**设计反思**：当前设计选择 DECIMAL 而非 DOUBLE PRECISION 作为坐标和性能数据的类型，在 100 倍规模下存储开销会更大（DECIMAL 变长存储），但气动工程中对精度要求高，权衡后精度优先。

## 问题三：性能数据是否适合列式数据库或分析型数据库？

**结论：`performance_record` 表非常适合列式存储，但当前系统的 OLTP + OLAP 混合场景更适合 PostgreSQL 行存储 + 物化视图的折中方案。**

**适合列式存储的理由**：

1. **查询模式以聚合为主**：核心查询如"同 Re 下多翼型 Cl/Cd 曲线对比"、"升阻比 TopN 排序"本质上是对少数列（`cl`、`cd`、`l_over_d`）的大量扫描，列存在此场景下 I/O 效率远超行存
2. **写入模式以批量追加为主**：性能数据通常批量导入，极少单条更新，适合列存的批量写入特性
3. **压缩率高**：`cl`、`cd` 等浮点列在列存中压缩率可达 5-10x

**但仍选择 PostgreSQL 行存储的原因**：

1. **外键约束需求**：`performance_record` 通过 `version_id` 和 `condition_id` 与版本表和工况表关联，列式数据库通常弱化或不支持外键
2. **事务一致性需求**：批量导入回滚（Scenario A）依赖行存的事务机制
3. **混合负载**：系统既要跑分析查询（OLAP），也要支持单条性能记录的 CRUD（OLTP）。DuckDB/ClickHouse 等列存引擎在单条更新场景下性能差
4. **当前规模**：22,907 行数据在 PostgreSQL 下完全够用，物化视图 `mv_airfoil_stats` 已覆盖首页统计需求

**如果规模扩大到百万级，推荐方案**：PostgreSQL 做主库（保证约束和事务） + DuckDB/ClickHouse 做分析副本（通过 `pg_cron` 定时同步），形成 OLTP+OLAP 分离架构。

## 问题四：版本表是独立设计更好，还是嵌入主表更好？

**结论：独立版本表更好。本系统采用 `airfoil_version` 独立表 + `is_current` 标记，这是正确的选择。**

**本系统设计**：

```
airfoil ──1:N── airfoil_version ──1:N── coordinate_point
                                    ──1:N── performance_record
```

**独立版本表 vs 嵌入主表对比**：

| 维度 | 独立版本表（当前） | 嵌入主表（version_no 列 + 覆写） |
|:-----|:------------------|:--------------------------------|
| **历史数据保留** | 旧版本保留原行，新增行 | 旧数据被 UPDATE 覆盖，不可追溯 |
| **坐标/性能关联** | 通过 `version_id` 外键精确关联 | 无法关联——坐标和性能失去版本归属 |
| **并发安全** | 新增版本是 INSERT，不锁旧版本 | UPDATE 需要行锁，并发下可能冲突 |
| **查询当前版本** | `WHERE is_current=true`（部分索引加速） | 直接查主表 |
| **版本对比查询** | `JOIN` 两个 version_id 即可 | 无法对比（旧版本已丢失） |
| **存储开销** | 多存历史版本行 | 只存当前版本 |

**为什么查询当前版本的额外 JOIN 代价可以接受？**

本系统通过 `v_current_airfoil_version` 视图 + `is_current` 部分索引，将 JOIN 代价降到最低。`EXPLAIN ANALYZE` 显示此查询为 Index Scan，耗时 <1ms。

**如果嵌入主表**：坐标点和性能记录将失去版本归属——你无法知道某条 Cl=1.2 的记录来自版本 1 还是版本 3。这对于工程数据管理是不可接受的。

**设计反思**：独立版本表的核心价值不是"多存几行"，而是**保证数据血缘可追溯**。在翼型工程中，一个翼型从初始设计到最终定型可能经历 5+ 次修订，每次修订的几何微调都影响性能——没有版本独立性，就无法回答"版本 2 的升阻比为什么比版本 1 高 5%？"

## 问题五：数据库与大模型协同的边界在哪里？

**结论：数据库负责约束校验与一致性保证，大模型负责语言理解与生成——二者不可互相替代。边界在"写入"操作上必须由数据库绝对控制。**

本系统通过实际 NL2SQL 评估实验（17 条用例）得出以下边界划分：

```
┌─────────────────────────────────────────────────────┐
│  大模型（DeepSeek）领域                               │
│  ✅ 自然语言 → SQL 翻译                               │
│  ✅ 查询结果的业务化解释                               │
│  ✅ 模糊问题的澄清与引导                               │
│  ⚠️ 列名推测（需 Schema 提示约束）                     │
│  ❌ 事务写入决策                                      │
│  ❌ 约束规则判断                                      │
├─────────────────────────────────────────────────────┤
│  数据库（PostgreSQL）领域                              │
│  ✅ 主键/外键/CHECK/UNIQUE 约束                       │
│  ✅ 索引优化与执行计划                                  │
│  ✅ 事务原子性、隔离性                                  │
│  ✅ 异常规则检测（SQL CHECK + 触发器）                  │
│  ❌ 自然语言理解                                      │
└─────────────────────────────────────────────────────┘
```

**实验证据**：

1. **大模型不能替代约束**：eval 中 DeepSeek 系统性将 `version_id` 简化为 `id`（P01/P12 weak 模式），如果没有数据库的 NOT NULL + 外键约束，这种错误 SQL 会静默返回错误结果或被拒绝执行。数据库约束是最后一道防线。

2. **大模型不能替代规则检测**：异常检测的 8 条规则（`negative_cd`、`cl_jump` 等）用 SQL CHECK + trigger 实现是确定性的，而用大模型做异常检测不仅慢，还会产生不确定性——NL2SQL 实验中 5/17 的 SQL 本身就有错误，让犯错概率 70% 的模型去检测异常，逻辑上自相矛盾。

3. **数据库不能替代大模型**：`api.list_airfoils_with_anomalies()` 返回 206 条异常记录，人可以读懂但需要时间。大模型在 E03 中对结果做了结构化解读——"NACA_0021 有 6 条异常提示，主要集中在负 Cd 和升阻比偏离"——这是数据库做不到的。

**边界原则**：大模型可以**读**（生成查询、解释结果），不能**写**（不能直接生成 INSERT/UPDATE/DELETE）。本系统的审计引擎 `audit.py` 强制执行这一原则——任何非 SELECT 语句一律 reject。

## 问题六：大模型是否适合直接参与事务写入？

**结论：绝对不适合。本系统的审计引擎明确禁止一切 DML 语句，这是经过实验验证的正确决策。**

**本系统的立场**：

`audit.py` 中硬编码了禁止列表：
```python
FORBIDDEN_KEYWORDS = ["INSERT", "UPDATE", "DELETE", "DROP", "ALTER", "TRUNCATE", "COPY"]
```

**理由**（结合实验）：

1. **NL2SQL 通过率仅 29.4%**：如果允许大模型写入，意味着 70% 的操作会导致数据错误或损坏。在读取侧，错误的 SQL 最多返回空结果或报错；在写入侧，一条 `UPDATE performance_record SET cd=-0.01 WHERE ...` 就能污染整个数据集。

2. **列名幻觉直接导致写错列**：实验中 weak 模式下 `av.id` 的错误如果发生在 UPDATE 语句中，可能静默更新错误的行（如果恰好有 `id` 列的话），比 SELECT 中的错误危险 100 倍。

3. **事务语义大模型无法理解**：批量导入回滚场景（Scenario A）需要"一条非法则整批回滚"——大模型不理解"原子性"概念，可能生成 `INSERT` 一批 + `COMMIT` + 再 `INSERT` 下一批的错误模式。

4. **大模型没有"后悔"机制**：数据库有 ROLLBACK，大模型生成的 SQL 一旦执行就不可逆（除非在事务中）。

**唯一可能的例外**：在人类审核通过后，大模型生成的写入 SQL 可以**作为建议**提交给人审批，但最终执行必须由人触发。本系统的 `nl2sql_audit` 表就是为此设计的——`audit_status='approved'` 是人工审核的结果，不是大模型自评。

**设计反思**：这不仅是技术问题，更是**责任归属问题**——如果大模型写入了错误数据，责任在谁？数据库的约束可以拦截一部分（如 Cd >= 0 CHECK），但无法拦截语义正确但值错误的写入（如把 Cl=1.2 写成 Cl=12.0）。写入的最终决定权必须留在人类手中。

## 总结

| 问题 | 本系统立场 |
|:-----|:----------|
| JSON vs 行存储 | **行存储**：约束、索引、外键、审计 > JSON 便利性 |
| 100 倍扩展 | **模式适用**，需补充分区和并发物化视图 |
| 列存 vs 行存 | **当前行存足够**，百万级后 OLTP+OLAP 分离 |
| 版本表独立 vs 嵌入 | **独立版本表**：数据血缘不可妥协 |
| 协同边界 | **数据库管约束，大模型管语言，读放行写禁止** |
| 大模型参与写入 | **绝对禁止**：29.4% 准确率下写入 = 数据灾难 |
